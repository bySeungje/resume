## 공통 서사 키워드

- **"지명받는 개발자"** — 한전KDN 3회, 우체국금융 2회 직접 호출
- **킬러 수치**: COBOL 70억 오류 발견, Spring Batch 80%+ 단축(월 175만 건), 마켓 연동 92% 단축(5개월→2~3주)
- **도메인 희소성**: AML + MyData + 결제정산 실무 경험 보유 개발자 국내 시장 1.5~3%

## 1. 한화시스템/ICT — 금융프로젝트 PL(파트리더)

### 1-1. 지원동기

11년간 금융·공공 도메인에서 개발자로 시작해 PL, PM, CTO까지 경험하며 깨달은 것이 있습니다. 금융 프로젝트의 성패는 기술력이 아니라 "규제와 업무를 이해하는 리더가 있느냐"에 달려 있다는 점입니다.

한전KDN에서 PL로 COBOL 레거시 전환을 이끌 때, 단순 마이그레이션이 아니라 전기공급약관과 코드의 불일치를 발견해 40년간 누적된 70억원 규모의 오류를 소급 보정했습니다. 기술 전환 프로젝트에서 도메인 규정까지 검증하는 PL은 흔하지 않다고 자부합니다.

한화시스템/ICT가 금융프로젝트 PL을 찾는다는 것은, 단순히 일정을 관리할 사람이 아니라 금융 규제를 이해하고 기술적 판단을 내릴 수 있는 리더를 원한다는 뜻이라고 이해했습니다. 그 역할에 가장 적합한 사람이 되고자 지원합니다.

### 1-2. 직무 적합성

**금융 프로젝트 PL/PM 실전 경험**

저는 금융 프로젝트에서 PL과 PM을 반복 수행한 경력이 있습니다. 한전KDN 영업배전 4.0 프로젝트(2023)에서는 PL로서 주니어 3명과 경력 6년차 2명, 총 5명의 개발자를 리드하며 COBOL→Spring Batch 전환을 완수했습니다. 단순히 일정 관리만 한 것이 아닙니다. Spring Batch 경험이 없는 팀원들에게 직접 기술 교육을 설계하고, 강의→코드리뷰→소단위 배치 배정→독립 작업의 단계적 위임 방식으로 팀 역량을 끌어올렸습니다. 결과적으로 월 175만 건 처리 시간을 3~4시간에서 40~50분으로 80% 이상 단축했습니다.

KISA 한국인터넷진흥원(2022~2023)에서는 PM·팀장으로 8개 사이트 운영을 총괄하며 신입 3명을 직접 채용하고, 매일 1~2시간 기술 교육과 수준별 업무 분배를 수행했습니다. 우체국 쇼핑몰(2018~2020)에서는 12명 프로젝트의 WBS 25%를 리드하며 외부 마켓 3사 공통 연동 아키텍처를 설계해 신규 마켓 온보딩을 92% 단축(5개월→2~3주)했습니다.

한전KDN에 3회, 우체국금융에 2회 직접 호출된 이력이 있습니다. "이 사람이 필요하다"고 지명받는 것은 금융 도메인 이해도와 프로젝트 리딩 능력을 현장이 인정한 결과라고 생각합니다.

### 1-3. 성장 가능성

솔직히 말씀드리면, Redis, Kafka, Kubernetes 등 모던 인프라 스택은 실무에서 직접 운영한 경험이 없습니다. 그러나 금융 프로젝트 PL의 핵심 역량은 인프라 도구의 숙련도보다 "금융 규제를 이해한 상태에서 기술적 판단을 내리는 능력"이라고 확신합니다. COBOL 70억 오류를 발견한 것도, AML 76개 룰의 DAG 의존관계를 재설계한 것도 도구가 아니라 도메인 이해에서 나온 성과입니다.

현재 개인 프로젝트(MeowRo)에서 Docker, CI/CD, Supabase(PostgreSQL) 기반 아키텍처를 직접 구축하며 모던 스택 경험을 확장하고 있으며, 입사 후에도 프로젝트에 필요한 기술을 빠르게 습득하겠습니다.

## 2. 코인원 — AML 시스템 개발자

### 2-1. 지원동기

AML 시스템을 "만든" 개발자는 있지만, "최적화한" 개발자는 극히 드뭅니다. 저는 우체국금융 AML 시스템에서 76개 STR(의심거래보고) 룰의 의존관계를 DAG로 분석하고, 4층 중첩 루프 구조를 진단하여 배치 처리 시간을 5시간에서 1시간 48분으로 64% 단축한 경험이 있습니다. 이 과정에서 EDD/CDD 워크플로우, 양방향 시제(Bi-temporal) 데이터 관리, 금융 규제 감사 대응 이력 추적 체계까지 직접 설계했습니다.

가상자산 거래소의 AML은 전통 금융과 규제 프레임워크는 같지만 거래 패턴이 다릅니다. 전통 금융 AML의 구조적 문제를 직접 경험하고 해결한 사람이, 가상자산이라는 새로운 도메인에서 더 효과적인 시스템을 만들 수 있다고 생각합니다. 코인원에서 그 확장을 해보고 싶습니다.

### 2-2. 직무 적합성

**JD 요구사항과의 직접 매칭**

코인원 AML 개발자 JD의 핵심 요구사항에 대해 저의 경험을 직접 대응하겠습니다.

**Java/Spring Boot**: 11년간 Java/Spring 기반 백엔드를 핵심 기술 스택으로 사용했습니다. Spring Boot + JPA, MyBatis, RESTful API 설계가 일상 업무입니다.

**AML 시스템 경험**: 우체국금융 AML에서 76개 STR 룰의 선후행 제약 룰과 병렬화 가능 룰을 분리하고, 룰 우선순위를 재설계했습니다. 단순히 룰을 구현한 것이 아니라, 기존 시스템의 구조적 병목을 진단하고 해결한 경험입니다. 룰 추출 모니터링 로그 시스템을 독자 구축하여 대규모 데이터 처리 중 오류를 실시간으로 탐지하는 체계도 마련했습니다.

**Spring Batch 우대**: 한전KDN에서 COBOL 정산 배치를 Spring Batch로 전환하며 청크 기반 멀티스레드, 지역본부별 파티셔닝, Flow 병렬화라는 3축 설계를 직접 수행했습니다. 월 175만 건 처리 시간 80%+ 단축이라는 정량적 성과가 있습니다.

**외부 API 연동**: 우체국 쇼핑몰에서 eBay·옥션·G마켓·11번가 4개 외부 마켓 API를 Anti-Corruption Layer + Adapter Pattern + Canonical Data Model로 연동했고, 한전KDN에서 카카오페이·네이버페이 PG 연동 정산 배치를 구현했습니다.

### 2-3. 성장 가능성

가상자산 AML은 전통 금융 대비 거래 속도가 빠르고 패턴이 복잡합니다. 이를 처리하려면 Redis 기반 실시간 캐싱이나 Kafka 기반 이벤트 스트리밍이 필요할 수 있다는 점을 알고 있습니다. 이 부분은 실무 경험이 없습니다.

그러나 AML의 본질은 인프라가 아니라 "의심 거래를 정확하게 탐지하는 룰 설계와 최적화"입니다. 76개 STR 룰을 DAG로 분석하고 최적화한 경험은 어떤 인프라 위에서든 적용 가능한 도메인 역량입니다. Redis와 Kafka는 입사 후 실무에서 빠르게 습득하겠지만, AML 도메인 이해도는 단기간에 만들 수 없는 자산입니다.

## 3. NHN PAYCO — 결제 플랫폼 서비스 개발자

### 3-1. 지원동기

결제 시스템을 코드로만 이해하는 개발자와, 비즈니스로 이해하는 개발자는 장애 대응에서 차이가 납니다. 저는 우체국 쇼핑몰에서 2년간 결제·클레임·정산 대사 처리 백엔드를 운영하며 일 17,000~26,000건의 주문을 처리했습니다. 복합 결제(신용카드·무통장·온누리상품권), 주문 취소, 환불, 정산 대사까지 결제의 전체 라이프사이클을 실무로 경험했습니다.

PAYCO는 단순 PG가 아니라 결제 플랫폼입니다. 다양한 결제 수단, 다양한 가맹점, 다양한 정산 주기를 하나의 시스템에서 처리해야 합니다. 저는 그 복잡성을 2년간 매일 운영하며 체감한 사람입니다. 결제 도메인을 코드가 아니라 비즈니스로 이해하는 개발자로서 PAYCO 플랫폼에 기여하고 싶습니다.

### 3-2. 직무 적합성

**결제/정산 실전 운영 2년**

우체국 쇼핑몰(2018~2020)에서 결제·클레임·정산 대사 처리를 담당하며, 수천만 건 규모의 주문아이템 데이터를 다루었습니다. 단순 CRUD가 아니라, 복합 결제 시 부분 취소 처리, 클레임(반품/교환/환불) 상태 전이, PG사별 정산 대사 로직 등 결제 도메인 특유의 복잡한 비즈니스 로직을 직접 구현하고 운영했습니다.

**PG 연동 경험**: 한전KDN에서 카카오페이·네이버페이 PG 연동 정산 배치를 구현했고, 우체국 쇼핑몰에서는 기존 결제 채널과 외부 마켓 결제를 통합 관리하는 아키텍처를 설계했습니다.

**대용량 처리**: 일 17K~26K 주문은 피크 시간대에 집중됩니다. 한전KDN Spring Batch에서 월 175만 건(피크 일 20~30만 건)을 처리한 경험과 합치면, 대용량 트랜잭션 처리에 대한 실전 감각이 있습니다.

**외부 시스템 연동**: eBay·옥션·G마켓·11번가 4개 외부 마켓을 Anti-Corruption Layer + Adapter Pattern + Canonical Data Model로 연동한 경험이 있습니다. 외부 시스템의 API 스펙 차이를 내부 표준 모델로 변환하는 설계는 결제 플랫폼에서 다양한 가맹점을 연동하는 구조와 본질적으로 동일합니다.

### 3-3. 성장 가능성

PAYCO 수준의 결제 플랫폼은 높은 가용성과 실시간 처리가 요구됩니다. Redis 기반 캐싱, 메시지 큐 기반 비동기 처리, 분산 트랜잭션 관리 등은 제가 실무에서 직접 운영해보지 않은 영역입니다.

그러나 결제 도메인의 핵심은 "정확성"입니다. 1원의 오차가 70억이 되는 것을 직접 발견한 사람으로서, 결제 시스템에서 정확성이 얼마나 중요한지 누구보다 잘 이해합니다. 인프라 도구는 학습할 수 있지만, 이 감각은 경험에서만 나옵니다. 입사 후 NHN의 기술 스택을 빠르게 습득하며, 결제 도메인 전문성으로 즉시 기여하겠습니다.

## 4. 트래블월렛 — 시니어 백엔드 (재무/정산)

### 4-1. 지원동기

정산 시스템의 가장 무서운 버그는 "아무도 모르게 누적되는 오차"입니다. 저는 한전KDN에서 COBOL 레거시 정산을 Spring Batch로 전환하는 과정에서, COBOL 내부 라운딩 로직이 전기공급약관 규정을 위반하고 있음을 발견했습니다. 건당 1원의 체계적 편향(systematic bias)이 40년간 한 방향으로만 누적되어 약 70억원의 오차를 만들었습니다.

정산의 정확성이 얼마나 중요한지, 그리고 그 정확성을 검증하는 일이 왜 시니어의 몫인지를 70억원을 통해 체감한 개발자입니다. 트래블월렛의 재무/정산 플랫폼은 다국적 통화, 환율 변동, 다양한 정산 주기가 얽힌 복잡한 도메인입니다. 그 복잡성 속에서 정확성을 지키는 시니어가 되고 싶어 지원합니다.

### 4-2. 직무 적합성

**정산 배치 전문가**

한전KDN 영업배전 4.0(2023)에서 COBOL 기반 정산 배치를 Spring Batch로 재설계·구현했습니다. 월 175만 건(피크 일 20~30만 건)을 처리하며, 청크 기반 멀티스레드 + 지역본부별 파티셔닝 + Flow 병렬화라는 3축 설계로 처리 시간을 3~4시간에서 40~50분으로 80% 이상 단축했습니다. 카카오페이·네이버페이·농어촌융자금 3개 외부 결제 채널의 정산까지 통합했습니다.

**결제/정산 운영 2년**: 우체국 쇼핑몰에서 일 17K~26K 주문의 결제·클레임·정산 대사 처리를 2년간 담당했습니다. 복합 결제 시 부분 취소, 반품/교환/환불 상태 전이, PG사별 정산 대사 로직 등을 직접 운영했습니다.

**금융 규제 이해**: AML 76개 STR 룰 최적화, 마이데이터 2.0 API 개발(은행·카드·보험·대출·지원 5개 권역), 양방향 시제(Bi-temporal) 데이터 관리 설계 등 금융 규제 컴플라이언스 경험이 풍부합니다. 트래블월렛의 외환 규제 대응에도 이 경험이 직접 연결됩니다.

**10년+ 경력의 시니어 리더십**: CTO 2회(널코드, 그리너랩), PM 2회(KISA 8개 사이트 총괄, 앤솔루션 개발팀 리딩), PL 1회(한전KDN 영배 4.0) 경험으로, 설계부터 팀 리딩까지 시니어로서의 역할을 수행해왔습니다.

### 4-3. 성장 가능성

트래블월렛의 정산은 다국적 통화와 실시간 환율이 개입하는 만큼, 제가 경험한 원화 기반 정산보다 복잡도가 높습니다. 또한 MSA 기반 분산 환경에서의 정산 일관성 보장은 제가 아직 실무에서 다루지 않은 영역입니다.

그러나 "정산 로직의 정확성을 검증하는 능력"은 통화가 바뀌어도 동일합니다. COBOL 라운딩 오류를 발견한 것은 Java나 Spring의 능력이 아니라, "이 숫자가 맞는가?"라는 질문을 끝까지 추적한 결과입니다. 분산 시스템의 기술적 학습과 병행하여, 정산 정확성에 대한 집요함으로 트래블월렛에 기여하겠습니다.

## 5. 핀다 — 시니어 백엔드 개발자

### 5-1. 지원동기

금융 서비스의 본질은 "사용자의 금융 문제를 기술로 해결하는 것"이라고 생각합니다. 저는 CTO로서 서비스를 제로부터 설계하고 출시한 경험이 있고, 동시에 AML·마이데이터·결제정산이라는 금융 핵심 도메인을 11년간 실무로 경험한 개발자입니다.

핀다는 대출 비교 플랫폼에서 시작해 종합 금융 플랫폼으로 확장하고 있습니다. 이 확장에는 금융 규제를 이해하면서 서비스를 설계할 수 있는 시니어가 필요합니다. 마이데이터 2.0 API를 은행·카드·보험·대출·지원 5개 권역에서 직접 개발한 경험, AML/KYC 규제 대응 경험, 결제정산 운영 경험을 핀다의 금융 서비스 확장에 활용하고 싶습니다.

### 5-2. 직무 적합성

**금융 도메인 전문성 — AML + MyData + 결제정산**

국내에서 AML, 마이데이터, 결제정산 세 영역을 모두 실무 경험한 개발자는 전체 시장의 1.5~3%에 불과합니다(10-Agent 분석 기준). 핀다의 금융 서비스가 확장될수록 이 세 도메인의 교차점에서 의사결정을 내릴 수 있는 시니어가 필요해질 것입니다.

- **마이데이터**: 우체국금융 마이데이터 2.0에서 5개 권역 신규 API 설계·개발, DB 컬럼명 메타시스템 표준 등록으로 SR 건수 20% 감소
- **AML/KYC**: 76개 STR 룰 DAG 분석 및 최적화, 양방향 시제 데이터 관리 설계
- **결제정산**: 일 17K~26K 주문 처리, 복합 결제, PG 연동, 정산 대사 2년 운영

**CTO 출신 서비스 리딩 경험**

널코드 CTO로서 3개 서비스를 기획부터 출시까지 전담했고, 그리너랩(트리닥터) CTO로서 농림축산부 나무의사 플랫폼을 제로부터 설계·구축했습니다. 사용자 인터뷰 기반 요구사항 분석, 기술 스택 선정, 아키텍처 설계, 팀 리딩까지 서비스 전체를 보는 시니어의 시각을 갖추고 있습니다.

**DB 아키텍처 경험**: Oracle OCP 11g 자격증 보유, 우체국 마이데이터 DB 표준화 수행, 한전KDN에서 COBOL 파일 I/O를 RDBMS로 전환하는 DB 재설계 경험이 있습니다.

### 5-3. 성장 가능성

핀다는 빠르게 성장하는 핀테크 기업으로, MSA, Kafka, Redis 등 모던 아키텍처를 적극 활용하고 있을 것으로 예상합니다. 이 기술 스택은 제가 실무에서 직접 운영한 경험이 없는 영역입니다.

그러나 CTO로서 기술 스택을 제로부터 선정하고 서비스를 출시한 경험이 있습니다. Flutter, Supabase, GitHub Actions 등 새로운 기술을 독학으로 습득하여 프로덕션에 적용한 학습 민첩성이 있습니다. 금융 도메인 전문성이라는 즉시 기여 가능한 자산 위에, 핀다의 기술 스택을 빠르게 흡수하여 "금융 사용자의 문제를 기술로 해결하는 CTO 출신 시니어"로 성장하겠습니다.
