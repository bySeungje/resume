## 2. 킬러 스토리 TOP 3 — STAR 형식

### Story #1: COBOL 40년 이자계산 오류 발견 (7/10 에이전트 선정)

**Situation**
한전KDN 영업배전 4.0 프로젝트(2023)에서 COBOL 레거시 정산을 Spring Batch로 전환하는 PL을 맡았다. 농어촌융자금은 장기 분할상환 상품으로, 매월 이자 계산 시 소수점 이하 금액이 발생한다.

**Task**
기존 COBOL 로직을 분석하여 Spring Batch로 정확하게 재구현해야 했다.

**Action**
데이터를 분석하던 중, 원금과 이자를 더한 값이 납부예정금액과 일치하지 않는 것을 발견했다. 깊이 파고들어 보니, 이건 COBOL vs Java 문제가 아니라 **COBOL 내부의 반올림 처리 로직 자체가 한전 전기공급약관("계산단위 미만의 끝수는 첫째자리에서 반올림")과 다르게 구현**되어 있던 문제였다.

건당 오차는 1원이지만, **체계적 편향(systematic bias)**이라 한 방향으로만 누적되었다.
- 농어촌 전기 고객 약 10만~수십만 호
- 매월 1원씩 편향 × 12개월 × 수년~수십 년
- 실제 확인 결과: **누적 오차 70억 원 이상**

운영하는 한전KDN에서는 "1원이 그 정도 차이를 만들 수 있냐"며 반발이 심했다. 한전과 KDN 직원이 모인 자리에서 **운영 데이터의 오차와 가입자 수를 추정하여 직접 시연회를 개최**하고, 데이터로 증명했다.

**Result**
40년간 누적된 데이터의 소급 보정을 실행. 농어촌융자금이 사회환원사업 성격이라 한전에서 손실을 흡수하는 것으로 정리되었지만, **코드 한 줄의 반올림 오류가 70억 원의 차이를 만든다는 사실**을 조직에 각인시켰다.

**면접에서의 한마디:**
> "시니어의 핵심은 '남들이 넘기는 1원의 차이를 추적해서 70억의 리스크를 발견하는 것'이라고 생각합니다."

### Story #2: Spring Batch 마이그레이션 — 월 175만 건 80%+ 단축

**Situation**
한전KDN 영업배전 4.0(2023). COBOL로 작성된 정산 배치가 월 175만 건(피크 일 20~30만 건)을 처리하는데 3~4시간이 걸렸다. COBOL JCL 구조에서는 병렬 처리를 하려면 JOB을 물리적으로 복사해야 했다.

**Task**
PL로서 COBOL→Spring Batch 전환을 설계하고, Spring Batch 경험이 없는 개발자 5명(주니어 3명 + 6년차 2명)을 교육해야 했다.

**Action**
3가지 축으로 개선을 설계했다:

1. **청크 기반 멀티스레드**: COBOL의 건별 순차 처리를 1,000건 단위 청크 + 8스레드로 전환 → 단순 계산만으로 8배 처리량 향상
2. **지역본부별 파티셔닝**: 한전은 수도권·강원·충청·호남·영남·제주 6개 지역본부 구조. 이 비즈니스 구조를 파티셔닝 키로 활용
3. **Flow 병렬화**: 순차 실행이 강제되던 JOB 체인을 독립적인 Step은 병렬 Flow로 재설계. 파일 I/O를 RDBMS로 전환하여 정렬·집계 병목 해소

팀 교육은 강의 → 공통 처리 코드리뷰 → 소단위 배치 배정 → 독립 작업의 단계적 위임으로 진행.
검증은 단건 비교 → 대량 비교 → 병행 운영 기간의 3단계로 수행.

**Result**
3~4시간 → 40~50분 (80%+ 단축). 카카오페이·네이버페이·농어촌융자금 3개 외부 결제 채널 연동 정산도 통합.

**면접에서의 한마디:**
> "기술만으로 8배를 만든 게 아닙니다. 한전의 지역본부 구조를 파티셔닝 키로 쓴 건 도메인을 아는 사람만 할 수 있는 설계입니다."

### Story #3: 외부 마켓 공통 아키텍처 → 연동 리드타임 92% 단축

**Situation**
우체국 쇼핑몰(2018~2020)에서 결제·클레임 백엔드를 운영하며 일 17K~26K 주문을 처리했다. eBay와의 첫 외부 마켓 연동에 5개월이 걸렸다.

**Task**
12명 프로젝트에서 WBS 25%를 리드하며, 후속 마켓(옥션·G마켓·11번가) 연동을 효율화하는 공통 아키텍처를 설계해야 했다.

**Action**
5개월이 걸린 근본 원인을 분석했다:
- 주문 테이블 구조 불일치 (외부 주문 별도 테이블 vs 통합 마이그레이션)
- 외부 마켓이 Push를 지원하지 않아 조회하지 않으면 주문을 알 수 없는 구조
- 물류 시스템과의 10분 단위 연계 제약

해결 구조:
- **Anti-Corruption Layer(부패 방지 계층)**: 외부 마켓마다 다른 주문·클레임 구조를 내부 표준 데이터 모델로 매핑
- **Adapter Pattern**: 마켓별 API 차이를 격리. 새 마켓 추가 시 Adapter만 구현하면 결제·취소·환불 등 비즈니스 로직 전체가 재사용
- **Polling 최적화**: 물류 연계 10분 제약 내에서 외부 주문 건수에 따라 3분까지 점진 조정

**Result**
11번가 연동 2~3주 만에 완료 (92% 단축). 이 아키텍처가 외부 마켓 매출 ~20% 증가에 직접 기여.

**면접에서의 한마디:**
> "5개월 걸린 원인이 '기술 난이도'가 아니라 '구조적 설계 부재'라는 걸 파악한 게 출발점이었습니다. Anti-Corruption Layer로 외부 마켓 차이를 격리하니, 새 마켓은 Adapter만 추가하면 되는 구조가 됐습니다."

## 4. 면접 필수 질문 & 완성 답변 10선

### Q1. "회사를 13개나 다니셨네요?"

> "SI 업계 특성상 수주 회사가 바뀌면 소속이 변경되지만, 개발자는 이어서 근무하는 구조입니다. 실제로 제 핵심 고객사는 5곳이고, 한전KDN은 3회, 우체국금융은 2회 재투입됐습니다.
>
> 특히 한전은 회사가 아니라 **저를 지명**해서 요청했고, 우체국금융은 쇼핑몰 근무 당시 꼼꼼한 검증과 문제 추적 능력을 인정받아 **금융팀 과장님이 직접 호출**한 케이스입니다.
>
> 같은 고객사에 반복 투입된다는 건, SI에서 신뢰의 가장 직접적인 증거입니다."

>
> 현재 개인 프로젝트에서 Redis와 Docker를 적용 중이며, GitHub에서 확인하실 수 있습니다."
